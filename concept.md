# Thursday

**Thursday** — локальный персональный ассистент для ноутбука в стиле Jarvis/Friday, но построенный не как один чат-бот, а как система управления компьютером, знаниями, файлами, голосом, Telegram и агентами разработки.

Главная идея: **Python Core управляет инструментами, памятью и правами доступа, а OpenCode/Codex используется как внешний инженерный мозг через hook/bridge-систему.**

> [!important] Принцип
> Thursday не должен быть просто “ИИ с микрофоном”. Он должен быть личной операционной оболочкой над ноутбуком.

## Зачем

- Быстро управлять ноутбуком голосом или через Telegram.
- Работать с Obsidian как с личной базой знаний.
- Запускать приложения, проекты, таймеры и сценарии.
- Давать OpenCode/Codex задачи по коду и файлам.
- Помнить контекст о пользователе, проектах, привычках и целях.
- Видеть экран через скриншоты, OCR и vision-модели.
- Отвечать голосом, текстом или уведомлением.

## Короткий образ системы

```text
Voice / Telegram / Hotkey
        ↓
Input Gateway
        ↓
Command Router
        ↓
Intent Parser
        ↓
Permission Layer
        ↓
Tools + Skills + Agent Bridge
        ↓
Files / Obsidian / Screen / Apps / Shell / OpenCode
        ↓
Response: Voice / Telegram / Notification / Obsidian Log
```

## Главные модули

### Python Core

Центральный процесс, который постоянно работает на ноутбуке.

Задачи:
- принимать команды;
- маршрутизировать намерения;
- проверять права доступа;
- вызывать tools;
- вести журнал действий;
- запускать OpenCode/Codex;
- возвращать ответ пользователю.

### Telegram Interface

Telegram нужен как удалённая панель управления.

Возможности:
- отправить команду текстом;
- получить результат;
- получить скриншот;
- подтвердить опасное действие;
- запустить задачу в проекте;
- получить уведомление о завершении;
- управлять ассистентом с телефона.

### Voice Interface

Голосовой режим нужен для быстрых команд.

Примеры:
- “Четверг, открой VS Code”;
- “Четверг, поставь таймер на 5 минут”;
- “Четверг, найди в Obsidian заметки про игру”;
- “Четверг, проверь проект через OpenCode”;
- “Четверг, что сейчас на экране?”

Рекомендуемый старт:
- сначала hotkey вместо постоянного wake word;
- затем faster-whisper;
- затем openWakeWord или Porcupine;
- затем голосовой ответ через `say`, Piper или ElevenLabs.

### Obsidian Memory

Obsidian — основная память и база знаний.

Память делится на:
- профиль пользователя;
- предпочтения;
- проекты;
- активные цели;
- журнал действий;
- важные факты;
- рабочие заметки;
- решения и выводы.

Принцип: ассистент не должен бесконтрольно записывать факты о пользователе. Лучше спрашивать:

```text
Я понял, что ты предпочитаешь Python для локальных агентов. Запомнить?
```

### File System Tools

Ассистент должен читать и создавать файлы, но не должен иметь безусловное право разрушать данные.

Безопасные действия:
- прочитать файл;
- найти файл;
- создать заметку;
- создать папку;
- показать список файлов.

Опасные действия:
- удалить файл;
- перезаписать файл;
- выполнить shell-команду;
- установить пакет;
- отправить приватные данные наружу.

Для опасных действий нужно подтверждение.

### Screen System

Экранный модуль нужен не для постоянного управления через картинку, а для понимания контекста.

Возможности:
- сделать скриншот;
- распознать текст через OCR;
- описать экран через vision-модель;
- найти активное окно;
- понять ошибку на экране;
- отправить скриншот в Telegram.

Минимальная версия:

```text
screenshot → OCR → краткое описание → ответ
```

### Apps System

Управление приложениями и проектами.

Команды:
- открыть VS Code;
- открыть Obsidian;
- открыть Terminal;
- открыть браузер;
- открыть конкретный проект;
- запустить dev-сервер;
- остановить процесс;
- показать активные процессы.

### OpenCode / Codex Hook Bridge

Это ключевая часть идеи.

Thursday не должен сам писать весь код. Он должен уметь передавать задачи агенту разработки.

Схема:

```text
User command
  ↓
Thursday Core
  ↓
Task Pack Builder
  ↓
OpenCode/Codex CLI
  ↓
Workspace changes
  ↓
Result summary
  ↓
Telegram / Voice / Obsidian log
```

Примеры задач:
- “проверь этот проект”;
- “найди баг”;
- “добавь функцию”;
- “сделай roadmap”;
- “обнови документацию”;
- “запусти тесты и скажи результат”.

Hook-система должна уметь:
- передавать задачу в OpenCode;
- прикладывать путь к проекту;
- добавлять контекст из Obsidian;
- сохранять результат в журнал;
- требовать подтверждение перед опасными действиями;
- отправлять краткий отчёт в Telegram.

## Режимы доступа

### Observe Mode

Только читает и объясняет.

Разрешено:
- смотреть файлы;
- читать Obsidian;
- делать скриншот;
- анализировать текст;
- отвечать пользователю.

### Assist Mode

Предлагает действия, но не меняет систему без разрешения.

Разрешено:
- подготовить план;
- предложить команды;
- создать черновик;
- показать, что будет изменено.

### Execute Mode

Выполняет безопасные действия.

Разрешено:
- открыть приложение;
- создать файл;
- создать заметку;
- поставить таймер;
- запустить заранее разрешённую команду.

### Admin Mode

Требует подтверждения.

Примеры:
- удалить файл;
- массово изменить заметки;
- запустить shell-команду;
- отправить данные наружу;
- установить зависимости;
- изменить настройки системы.

## Первый рабочий сценарий

```text
Пользователь пишет в Telegram:
“Четверг, открой проект Thursday и создай заметку с планом MVP”

Thursday:
1. Проверяет команду.
2. Видит, что это безопасное действие.
3. Создаёт заметку в Obsidian.
4. Открывает проект.
5. Отправляет ответ в Telegram.
6. Записывает действие в журнал.
```

## Второй рабочий сценарий

```text
Пользователь говорит:
“Четверг, проверь экран”

Thursday:
1. Делает скриншот.
2. Прогоняет OCR.
3. Если есть ошибка, кратко объясняет.
4. Спрашивает, нужно ли отправить задачу в OpenCode.
```

## Третий рабочий сценарий

```text
Пользователь пишет:
“Проверь проект wald-ai и исправь баги”

Thursday:
1. Определяет, что это coding-task.
2. Готовит task pack.
3. Передаёт задачу в OpenCode/Codex.
4. Ждёт завершения.
5. Получает summary.
6. Отправляет отчёт в Telegram.
7. Сохраняет результат в Obsidian.
```

## Чего не делать на старте

- Не делать постоянное фоновое прослушивание сразу.
- Не давать полные права на удаление файлов.
- Не строить сложную локальную LLM с первого дня.
- Не пытаться сделать идеального Jarvis за один этап.
- Не делать vision главным способом управления.

## Что сделать первым

1. Создать Python-проект.
2. Поднять Telegram bot.
3. Добавить config.
4. Добавить command router.
5. Добавить tools: apps, timers, obsidian, files.
6. Добавить permissions.
7. Добавить action log.
8. Добавить OpenCode bridge.
9. Добавить voice output.
10. Потом добавить voice input.

